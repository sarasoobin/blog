# TanStack Query, 왜 쓰고 어떻게 쓰는가

React에서 서버 데이터를 다룰 때 사실상 표준으로 자리 잡은 라이브러리, TanStack Query(구 React Query)에 대해 정리한다. 도입 이유부터 사전 지식, 주의할 점, 기본 사용법까지 다룬다.

---

## 1. TanStack Query를 왜 쓰는가

기존 React에서 서버 데이터를 가져올 때는 `useState` + `useEffect` + `fetch` 조합을 사용해 컴포넌트마다 비슷한 패턴을 반복해 작성해야 했다. 이 방식에는 다음과 같은 한계가 있다.

- **캐시가 없다**: 같은 데이터를 보는 화면을 다시 들어갈 때마다 매번 서버에 요청한다.
- **중복 요청이 합쳐지지 않는다**: 한 화면에서 여러 컴포넌트가 같은 데이터를 부르면 동일한 요청이 여러 번 나간다.
- **데이터 신선도 관리가 없다**: 데이터가 언제 받은 것인지 추적하지 않으므로 갱신 정책을 세우기 어렵다.
- **부가 기능을 모두 직접 구현해야 한다**: 자동 재시도, 윈도우 포커스 갱신, 요청 취소 등을 일일이 짜야 한다.

TanStack Query는 이 모든 문제를 자동화한다. 개발자가 이 데이터가 필요하다고 선언하기만 하면 캐싱, 동기화, 재시도, 신선도 관리는 라이브러리가 처리한다. 결과적으로 코드가 단순해지고, 서버 부하가 준다.

---

## 2. 사용 전 알아야 할 개념

### 서버 상태 vs 클라이언트 상태

| 구분 | 클라이언트 상태 | 서버 상태 |
|---|---|---|
| 예시 | 사이드바 열림, 다크모드, 폼 입력값 | 게시글 목록, 사용자 정보, 좋아요 수 |
| 주인 | 브라우저 | 서버 |
| 도구 | `useState`, `useReducer` | TanStack Query |

TanStack Query는 **서버 상태**만을 위한 도구다. 클라이언트 상태는 여전히 `useState`로 관리한다.

### 비동기와 Promise

서버 요청은 결과가 즉시 오지 않는 비동기 작업이다. `async/await` 또는 `.then()` 문법을 사용한다.

### fetch / axios

TanStack Query는 데이터를 직접 가져오지 않는다. 사용자가 정의한 비동기 함수(보통 `fetch` 또는 `axios`)를 받아서 그 위에 캐싱과 상태 관리를 얹는다.


---

## 3. 사용 시 주의할 점

### queryKey에 의존성을 빠뜨리지 않는다

데이터가 특정 입력값(사용자 ID, 정렬 옵션, 검색어, 좌표 등)에 의존하면 그 입력값을 반드시 `queryKey`에 포함시켜야 한다. 빠뜨리면 다른 입력 환경의 캐시가 잘못 표시되는 사고가 발생한다.

```ts
// 사용자 ID에 의존하므로 키에 포함
useQuery({ queryKey: ['user', userId], queryFn: ... })

// 정렬 옵션에 의존
useQuery({ queryKey: ['posts', sort], queryFn: ... })
```

### QueryClient는 컴포넌트 함수 바깥에서 생성한다

함수 안에서 `new QueryClient()`를 호출하면 리렌더링마다 새로운 인스턴스가 생성되어 캐시가 매번 초기화된다.

### staleTime과 gcTime의 관계를 지킨다

신선도 유효 기간(`staleTime`)이 메모리 보관 기간(`gcTime`)보다 길면 캐시가 신선해질 새도 없이 삭제될 수 있다. 일반적으로 `gcTime ≥ staleTime`으로 설정한다.



### 변경 요청에는 자동 재시도를 적용하지 않는다

조회 요청은 자동 재시도가 안전하지만, 생성·수정·삭제 같은 변경 요청은 자동 재시도 시 중복 처리될 위험이 있다. 결제가 두 번 일어나거나 같은 글이 두 번 등록될 수 있으므로 변경 요청 설정에서는 `retry: false`로 설정하는 것이 일반적이다.

```ts
const queryClient = new QueryClient({
  defaultOptions: {
    queries: { staleTime: 1000 * 60 * 5 },
    mutations: { retry: false },
  },
});
```

## 4. 사용 방법

### 4-1. 설치

```bash
npm install @tanstack/react-query
# 또는
pnpm add @tanstack/react-query
```

### 4-2. QueryClient와 Provider 설정

```tsx
// App.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

const queryClient = new QueryClient();

function App() {
  return (
    <QueryClientProvider client={queryClient}>
      {/* 모든 컴포넌트 */}
    </QueryClientProvider>
  );
}
```

### 4-3. useQuery로 데이터 가져오기

```tsx
const { data, isPending, isError, error } = useQuery({
  queryKey: ['todos'],
  queryFn: fetchTodos,
});

if (isPending) return <Loading />;
if (isError) return <Error message={error.message} />;
return <TodoList data={data} />;
```

`queryKey`는 캐시를 식별하는 키이며, 같은 키를 사용하는 컴포넌트는 캐시를 공유한다. `queryFn`은 Promise를 반환하는 비동기 함수다.

### 4-4. 캐시 정책 설계: staleTime과 gcTime

| 옵션 | 의미 | 비유 |
|---|---|---|
| `staleTime` | 데이터를 신선한 것으로 간주하는 시간 | 우유 유통기한 |
| `gcTime` | 비활성 캐시의 메모리 보관 기간 | 안 쓸 때 보관 기간 |

데이터 특성별 권장 설정.

| 데이터 종류 | staleTime | gcTime |
|---|---|---|
| 거의 변하지 않음 (카테고리, 국가) | 10~30분 | 1시간 이상 |
| 가끔 변함 (게시글, 프로필) | 1~5분 | 10~30분 |
| 자주 변함 (알림, 좋아요 수) | 0~10초 | 1~5분 |

### 4-5. 조건부 요청: enabled

특정 조건이 충족되어야 요청을 보낼 때 사용한다.

```tsx
useQuery({
  queryKey: ['user', userId],
  queryFn: () => fetchUser(userId),
  enabled: !!userId, // userId가 있을 때만 요청
});
```

### 4-6. Devtools 활용

개발 모드에서 쿼리 상태와 캐시를 시각적으로 확인할 수 있는 디버깅 도구를 제공한다.

```tsx
import { ReactQueryDevtools } from '@tanstack/react-query-devtools';

<QueryClientProvider client={queryClient}>
  {/* 컴포넌트 */}
  <ReactQueryDevtools initialIsOpen={false} />
</QueryClientProvider>
```

각 쿼리의 신선도(fresh/stale), 상태(fetching/success/error), 캐시된 데이터를 모두 화면에서 확인할 수 있어 디버깅에 유용하다.

### 4-7. 무한 스크롤: useInfiniteQuery

페이지 단위로 데이터를 누적해 가져올 때 사용한다.

```tsx
const { data, fetchNextPage, hasNextPage, isFetchingNextPage } = useInfiniteQuery({
  queryKey: ['posts'],
  queryFn: ({ pageParam }) => fetchPosts(pageParam),
  initialPageParam: 1,
  getNextPageParam: (lastPage, allPages) =>
    lastPage.length < PAGE_SIZE ? undefined : allPages.length + 1,
});
```

핵심 개념은 다음과 같다.

- `initialPageParam`: 첫 페이지 요청 값
- `getNextPageParam`: 다음 페이지 파라미터를 계산하는 함수. `undefined`를 반환하면 종료
- `data.pages`: 받아온 페이지들이 누적된 배열
- `fetchNextPage`: 다음 페이지를 가져오는 함수
- `hasNextPage`: 다음 페이지 존재 여부
- `isFetchingNextPage`: 다음 페이지를 가져오는 중인지 여부

브라우저 기본 API인 **Intersection Observer**와 함께 사용하면 스크롤이 끝에 닿는 시점에 자동으로 다음 페이지를 가져오는 무한 스크롤을 구현할 수 있다.



### 4-8. Skeleton UI로 로딩 경험 개선

데이터 도착 전 빈 화면 대신 실제 콘텐츠의 윤곽을 회색 박스로 보여주는 기법이다. 사용자가 "멈췄나?"라고 불안해하지 않게 하고, 데이터 도착 시 화면 레이아웃이 흔들리지 않도록 도와준다.

`isPending`(초기 로딩)과 `isFetchingNextPage`(추가 로딩) 상태에 따라 적절한 위치에 Skeleton을 노출한다. 단, 응답이 매우 빠른 경우 Skeleton이 잠깐 보였다 사라지면 깜빡임으로 느껴질 수 있어, 일정 시간 이상의 지연에만 노출하도록 조정하는 것이 좋다.

## 5. 마치며

TanStack Query의 본질은 **서버 상태를 캐싱·동기화·재시도·신선도 관리까지 자동화하는 것**이다. 

**<핵심개념>**
- **queryKey**: 캐시의 식별자. 의존하는 입력값을 모두 포함시킨다.
- **staleTime**: 데이터의 신선도 유효 기간.
- **gcTime**: 비활성 캐시의 메모리 보관 기간.
- **useInfiniteQuery + Intersection Observer**: 무한 스크롤 구현의 표준 조합.

페이지마다 반복되던 데이터 패칭 코드를 단순화하고 캐시 정책을 명시적으로 설계할 수 있다는 점에서, 현대 React 프로젝트의 사실상 표준이라 할 수 있다.

## 6. 퀴즈


### Q1. TanStack Query에 대한 설명으로 옳지 않은 것은?

① 서버 상태(server state)를 관리하기 위한 라이브러리다.
② 데이터를 가져오는 fetch나 axios를 대체한다.
③ 캐싱, 자동 재시도, 신선도 관리를 자동으로 해준다.
④ 같은 `queryKey`를 사용하는 컴포넌트끼리 캐시를 공유한다.

<details>
<summary>정답 보기</summary>

**정답: ②**

TanStack Query는 fetch나 axios를 **대체하지 않는다**. 데이터를 직접 가져오지 않고, 사용자가 정의한 비동기 함수(`queryFn`) 위에 캐싱·상태 관리·동기화 기능을 얹어주는 도구다. 즉 fetch나 axios는 그대로 사용하되, 그 호출을 `useQuery` 안에서 래핑하는 구조다.

나머지 보기는 모두 맞는 설명이다. ①은 라이브러리의 본질, ③은 핵심 가치, ④는 `queryKey`의 캐시 공유 동작에 해당한다.

</details>

### Q2. 다음 코드에는 버그가 있다. 어떤 문제이며, 어떻게 고쳐야 하는가?

```tsx
function UserProfile({ userId }) {
  const { data } = useQuery({
    queryKey: ['user'],
    queryFn: () => fetch(`/api/users/${userId}`).then(r => r.json()),
  });

  return <div>{data?.name}</div>;
}
```

<details>
<summary>정답 보기</summary>

**정답**

`queryKey`에 의존성인 `userId`가 빠져 있다. `userId`가 바뀌어도 키는 항상 `['user']`로 동일하기 때문에, TanStack Query는 "이미 캐시에 있는 데이터"라고 판단해 이전 사용자의 정보를 그대로 보여준다.

**수정 후**

```tsx
queryKey: ['user', userId],
```

`queryFn` 내부에서 사용하는 변수는 `queryKey`에 모두 포함시켜야 한다. 그래야 변수가 바뀔 때마다 다른 캐시 항목으로 분리되어 정확한 데이터를 가져온다.

규칙

> **queryFn에서 쓰는 변수는 queryKey에도 똑같이 들어가야 한다.**

</details>

### Q3. `staleTime`과 `gcTime`에 대한 설명으로 옳은 것은?

① `staleTime`은 캐시가 메모리에서 삭제되기까지의 시간이다.
② `gcTime`은 데이터가 신선한 것으로 간주되는 시간이다.
③ `gcTime`은 일반적으로 `staleTime`보다 길거나 같게 설정한다.
④ 두 값은 같은 의미이며 어느 쪽을 써도 동작이 동일하다.

<details>
<summary>정답 보기</summary>

**정답: ③**

| 옵션 | 의미 | 비유 |
|---|---|---|
| `staleTime` | 데이터를 신선한 것으로 간주하는 시간 | 우유 유통기한 |
| `gcTime` | 비활성 캐시의 메모리 보관 기간 | 안 쓸 때 보관 기간 |

- ①은 두 옵션을 혼동한 설명이다. 메모리 삭제 시점은 `gcTime`이 결정한다.
- ②도 정의가 뒤바뀐 설명이다. 신선도는 `staleTime`의 영역이다.
- ④는 틀렸다. 두 값은 의미와 동작이 완전히 다르다.

`gcTime ≥ staleTime` 관계를 지키는 이유는, 신선한 동안 캐시가 살아있어야 의미가 있기 때문이다. 만약 `gcTime`이 더 짧으면 데이터가 신선해질 새도 없이 메모리에서 사라져 캐시 효과가 없어진다.

</details>
